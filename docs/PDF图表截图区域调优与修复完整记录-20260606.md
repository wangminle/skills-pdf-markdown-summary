## 1. 本次任务目标

本次工作的目标不是只让两个 PDF “能够导出图片”，而是重新梳理截图区域的完整决策链路，并使用画线调试输出逐阶段验证：

1. 图表 caption 是否定位正确。
2. 图表位于 caption 上方还是下方。
3. baseline、文本裁切、对象裁切、版式裁切、像素裁切和最终采用区域是否合理。
4. 最终截图是否完整保留目标图表。
5. 最终截图是否混入正文、章节标题、相邻图表或 caption。
6. 调整是否形成通用算法，而不是仅针对两个 PDF 写死参数。

最终要求：

- `DeepSeek_V3_2.pdf` 正确导出 4 个 Figure 和 1 个完整 Table。
- `FunAudio-ASR.pdf` 正确导出 4 个 Figure 和 8 个完整 Table。
- 所有最终画线区域合理有效。
- 所有导出图片完整、纯净，不混入正文和章节标题。

## 2. 初始流程与主要问题

原始截图流程已经包含较多阶段：


流程组件并不缺少，但多个阶段之间缺少明确的优先级和保护关系，导致前面阶段得到的正确结果可能被后续阶段覆盖。

- `--debug-visual` 参数虽然开启，但没有真正生成画线图片。
- Table caption 通常位于表格下方，算法却可能默认向下截图。
- 无矢量边框的表格缺少方向判断证据。
- 正确精裁结果可能因高度或面积比例过小而回退到错误 baseline。
- 表格行可能被通用正文检测误判。
- 章节标题可能被误认为图表内容。
- DeepSeek Table 1 顶部表头和部分数据行被裁掉。


## 4. 第一阶段：修复画线调试链路

### 4.1 发现的问题

开启 `--debug-visual` 后没有画线输出，原因并不是命令参数未传入，而是调试链路内部存在多个兼容性问题：

1. 调用方使用了错误的 `DebugStageInfo` 字段：

   ```python
   DebugStageInfo(stage=..., rect=...)
   ```

   实际数据结构需要的是：

   ```python
   DebugStageInfo(name=..., bbox=..., color=..., description=...)
   ```

2. 调试渲染使用了当前 PDF 后端包装层不支持的 API：

   - `page.parent`
   - `page.get_pixmap(matrix=...)`

3. 生成的调试文件没有写入 `AttachmentRecord.debug_artifacts`。

4. 被拒绝的 Table 候选没有留下画线证据，难以判断拒绝是否合理。

### 4.2 修复内容

- 新增统一的 `create_debug_stage()`。
- 调试渲染改用后端兼容的 `page.get_pixmap(dpi=...)`。
- 去除对 `page.parent` 的依赖。
- Figure/Table 保存画线图后，将路径写入 `debug_artifacts`。
- Table 被拒绝时也生成 `debug_rejected` 画线输出和拒绝原因图例。

### 4.3 结果

从 `tests/results/20260606-002/` 开始，画线图能够稳定生成，后续所有算法问题都可以定位到具体阶段。

这一步是本次调优能够成功的基础。如果没有可靠的画线输出，后续只能盲调参数，无法判断正确区域究竟在哪一步被破坏。

## 5. 第二阶段：重新设计 Table 方向判断

### 5.1 根因

两个目标 PDF 中，Table caption 都位于表格下方，正确截图方向应为 `above`。

旧流程的问题：

- Table 的默认或全局方向容易选择 `below`。
- 很多论文表格只有文字和少量横线，没有可用的矢量对象证据。
- `score_direction_for_caption()` 使用了不兼容的位图渲染调用，异常被捕获后方向证据实际为零。

### 5.2 调整内容

Table 局部方向判断加入文本结构证据：

- 短文本行或单元格文本的比例。
- 宽长正文行的比例。
- 文本与 caption 的距离。
- caption 上下两侧的可用空间。
- 矢量和图片对象证据作为补充，而不是唯一依据。

同时修复方向评分中的 PDF 渲染调用。

### 5.3 结果

从 `tests/results/20260606-003/` 开始：

- DeepSeek Table 1 可以被识别并导出。
- FunAudio Table 数量由初始 4 个提升到 6 个。

此时方向问题基本解决，但表格验收和区域纯净度仍有问题。

## 6. 第三阶段：修复短图和章节标题污染

### 6.1 根因

FunAudio Figure 3 是高度较短的流程图，其上方紧邻章节标题：

```text
4.1 Pre-training of Audio Encoder
```

旧版式裁切存在两个问题：

- 固定最小高度约束过大，短图无法充分收紧。
- 标题块还需要满足正文式宽度门槛，导致明显章节标题没有被当作远端边界。

### 6.2 调整内容

- 标题块可直接作为远端裁切边界。
- 短内容最小高度限制由固定约束调整为更合理的下限。
- 新增短 Figure 紧邻章节标题的回归测试。

### 6.3 结果

FunAudio Figure 3 不再混入章节标题，短图仍保持完整。

## 7. 第四阶段：修复 Table 验收与回退

### 7.1 根因

Table 的正确精裁结果通常比 520pt baseline 小很多。旧流程会认为：

```text
height_ratio 或 area_ratio 过小
→ 精裁结果不可信
→ 回退到 Phase A 或 baseline
```

回退后的区域往往重新混入正文，或者导致正确 Table 被拒绝。

同时，通用正文污染检测看到表格中的大量文字时，也可能把表格误认为正文。

### 7.2 调整内容

新增 `looks_like_table_text()`，从文本结构上区分：

- 表格：短文本和单元格文本为主，宽长正文行占比较低。
- 正文：大量横跨候选区域的长文本行。

验收调整：

- 对已确认具有表格结构的精裁结果，低高度比和低面积比改为软限制。
- 过窄区域仍然硬拒绝。
- 表格结构成立时，不因通用正文污染检测而拒绝。
- 被拒绝的候选生成调试画线输出。

### 7.3 结果

在 `tests/results/20260606-004/` 中：

- DeepSeek：4 Figure + 1 Table。
- FunAudio：4 Figure + 8 Table。

数量已完整，但 FunAudio Table 1 仍混入表格前正文，DeepSeek Table 1 仍不完整。

## 8. 第五阶段：远端正文迭代清理

### 8.1 根因

FunAudio Table 1 前方有连续多行正文。旧的远端正文清理只执行一次，每次只清理边缘附近少量正文，因此无法一次去除完整正文段落。

### 8.2 调整内容

新增 `trim_far_side_text_iterative()`：

```text
扫描远端正文
→ 裁掉一段
→ 重新扫描新边缘
→ 继续裁剪
→ 遇到非正文内容后停止
```

设置有限迭代次数，避免无限收缩。

### 8.3 结果与局限

该方法能够逐步清理正文，但同时暴露出更深层问题：

- 表格顶部的宽文字表头也可能被当成正文。
- DeepSeek Table 1 的表头被误裁。
- 单靠通用正文清理无法稳定区分表格和正文。

因此后续不再继续调整正文清理参数，而是设计表格专用的结构识别流程。

## 9. 第六阶段：设计连续表格行带识别

### 9.1 设计目标

需要从 caption 一侧反向识别真正连续的表格区域：

```text
caption
↑
最近的表格行
↑
连续表格行
↑
表头或稀疏分类标签
↑
正文或章节标题，停止
```

### 9.2 新增算法

新增 `refine_clip_to_table_band()`：

1. 收集候选区域内的文本行。
2. 按 Y 坐标将文本合并为视觉行。
3. 将同一视觉行中分离的文字块视为单元格。
4. 从 caption 一侧开始，寻找最近的多单元格行。
5. 沿远端持续扩展连续表格行。
6. 允许少量短稀疏分类标签，例如 DeepSeek Table 1 左侧的：

   ```text
   General
   Search Agent
   Code Agent
   Math
   ```

7. 遇到正文或章节标题后停止。
8. 使用识别出的表格行带收紧 Y 边界。

### 9.3 为什么不能只调参数

FunAudio Table 1 和 DeepSeek Table 1 的冲突非常典型：

- FunAudio Table 1 需要激进裁掉表格前正文。
- DeepSeek Table 1 需要完整保留表头和稀疏分类标签。

如果仅调整正文宽度、行数或高度阈值：

- 阈值激进时，DeepSeek 表头被裁掉。
- 阈值保守时，FunAudio 正文无法去除。

因此必须从“正文清理”升级为“表格结构识别”。

## 10. 第七阶段：处理流程阶段互相覆盖

连续表格行带算法加入后，结果最初没有立即生效。通过画线图和阶段坐标，依次定位到多个覆盖点。

### 10.1 第一次覆盖：算法执行太晚

问题：

- 表格行带在像素 autocrop 后执行。
- DeepSeek 表头已经被更早阶段裁掉，后续无法恢复。

调整：

- 将表格行带识别前移到像素 autocrop 之前。

### 10.2 第二次覆盖：通用最小高度回退

问题：

- FunAudio Table 1 的正确表格区域高度只占 baseline 很小比例。
- 即使表格行带识别正确，通用 `35%` 最小高度门槛仍会回退到错误区域。

调整：

- 已确认具有表格结构的行带允许绕过通用最小高度回退。

### 10.3 第三次覆盖：版式裁切先删除表头

问题：

- DeepSeek 表头在表格行带识别之前被版式或 X 向阶段裁掉。

调整：

- 表格行带始终基于原始 `base_clip` 识别完整 Y 边界。
- 最终只将行带识别出的 Y 边界合并回当前 X 边界。
- 保留列感知的 X 向精裁，同时恢复完整表头。

### 10.4 第四次覆盖：远端正文清理再次裁掉表头

问题：

- 连续表格行带已经恢复 DeepSeek 表头。
- 旧远端正文清理又把宽表头误认为正文并裁掉。

调整：

- 已确认连续表格行带后，不再执行通用远端正文迭代清理。

### 10.5 第五次覆盖：远端正文证据限制再次裁掉列标题

问题：

- DeepSeek 顶部列标题行仍被 `detect_far_side_text_evidence()` 当作正文边界。

调整：

- 已确认连续表格行带后，同时绕过通用远端正文证据限制。

这些问题说明，本次最终效果改善的核心不是某一个参数，而是明确了流程优先级：

```text
表格结构证据
优先于
通用正文清理、通用版式裁切和通用比例回退
```

## 11. 第八阶段：排除两段式章节标题

### 11.1 新问题

FunAudio Table 4 上方的章节标题由两个文本块组成：

```text
6.2.3
Evaluation on Noise Robustness
```

它被初版表格行带识别为“两单元格表格行”，导致章节标题被带入截图。

### 11.2 调整内容

对恰好只有两个单元格的视觉行增加宽度约束：

- 两个单元格必须横跨候选区域的大部分宽度，才视为表格行。
- 窄的两段式标题不视为表格行。
- 三个及以上单元格的视觉行仍直接视为表格结构证据。

新增对应回归测试，确保：

- FunAudio 的两段式章节标题被排除。
- DeepSeek 的宽表头仍完整保留。

## 13. 最终流程

### 13.1 Figure 最终流程

```text
高分 caption 定位
→ 局部方向和全局方向判断
```

### 13.2 Table 最终流程

```text
高分 caption 定位
→ 基于文本结构判断上下方向
→ 同页相邻 caption 限制 baseline
→ 通用文本、对象和 X 向裁切
→ 基于原始 baseline 识别连续表格行带
→ 将表格行带 Y 边界合并回当前 X 边界
→ 已确认表格行带时，跳过会破坏表头的通用正文清理
→ 像素 autocrop
→ 表格文本结构验收
→ 仅对非表格结构执行严格回退或拒绝
→ 保存图片与画线产物
```

## 16. 为什么本次最终效果较好

本次效果明显改善的主要原因有四点。

### 16.1 从盲调参数转为逐阶段可观察

修复画线调试链路后，可以明确看到：

- 正确区域在哪一步产生。
- 哪一步裁掉了表头。
- 哪一步把正确精裁回退到错误 baseline。
- 哪一步把章节标题当成表格。

因此每次调整都有明确根因，不再依赖猜测。

### 16.2 从通用裁剪升级为对象类型专用结构识别

Figure 和 Table 的结构不同：

- Figure 更依赖图片、矢量对象和空白边界。
- Table 更依赖文本行、单元格排列和连续行带。

本次为 Table 增加专用的连续行带识别，解决了通用正文清理无法同时兼顾 FunAudio 和 DeepSeek 的矛盾。

### 16.3 明确流程优先级

旧流程的问题不是没有算法，而是多个阶段互相覆盖。

最终明确：

```text
可靠的表格结构证据
优先于
通用正文清理、通用版式裁切、通用比例阈值和回退
```

这样正确结果不会在后续阶段被破坏。

### 16.4 每个暴露问题都补充回归测试

本次不是只检查最终图片。每个主要根因都增加了小型失败用例，确保后续修改不会轻易恢复旧问题。
